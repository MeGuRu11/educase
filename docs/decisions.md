# EduCase — Журнал архитектурных решений (ADR)

> ⚠️ **РЕКОНСТРУКЦИЯ ИЗ ПАМЯТИ** ассистента о прошлых сессиях, а НЕ оригинальный файл.
> Сверь каждый пункт с тем, что реально согласовано с кафедрой, и поправь.
> По номеру уверенно подтверждён только **ADR-003**; нумерация и заголовки остальных —
> моё восстановление по смыслу. Статусы проставь сам.

## Контекст проекта

EduCase — **третий заход**. Два прошлых провалились из-за over-documentation и scope creep.
Главный принцип: **скорость до работающего кода**, без артефактов, которые придётся
переделывать. Дедлайн: **25 июня 2026**.

Среда исполнения: компьютерный класс ВМА им. Кирова, **Windows 10/11**. Оборудование
модернизируется — прежние ограничения (совместимость с Win7, слабое железо) больше не
действуют.

Стек: **Python 3.12 + PySide6, SQLAlchemy 2 + SQLite, PyInstaller, loguru, pytest-qt.**

Важное уточнение по контексту: «вариант от руководителя» был **примером кейса (sample case)**,
а не спецификацией проекта — это различие пришлось проговаривать отдельно.

---

## ADR-001 — Две раздельные программы

**Решение.** Не одно приложение с режимами, а **две самостоятельные программы**:
- **Constructor** — для преподавателя. Без авторизации, работает как инструмент.
- **Player** — для курсанта. Загружает кейс, проходит, формирует результат.

**Обоснование.** Разные роли, разный жизненный цикл, проще раздавать в класс. Авторизация
не нужна — это не многопользовательская система.

**Последствия.** Общий доменный слой и кодеки выносятся в общий пакет (`educase_core`),
два GUI-приложения собираются в два EXE.

---

## ADR-002 — Обмен только файлами (.educase / .eduresult)

**Решение.** Обмен между Constructor и Player — через файлы-архивы:
- `.educase` — кейс (преподаватель → курсант);
- `.eduresult` — результат прохождения (курсант → преподаватель).

Формат — **ZIP-контейнер** с внутренним JSON + ассеты (фото, документы). **JSON никогда не
показывается пользователю** — это внутреннее представление. Перенос архивов — **вручную через
Проводник**.

**Обоснование.** Класс без общей сети/сервера; простой и надёжный способ раздачи и сбора.

---

## ADR-003 — Сетевого кода нет (подтверждён)

**Решение.** В проекте **запрещён любой сетевой код**: никаких HTTP-клиентов/серверов,
`requests`/`httpx`/`socket`, синхронизации по сети. Всё оффлайн.

**Обоснование.** Сетевая сложность была повторяющейся ловушкой в планировании; этот ADR создан
специально, чтобы закрыть вопрос навсегда. Обмен решается файлами (ADR-002).

**Последствия.** Любое предложение, тянущее за собой сеть, отклоняется без нового ADR.

---

## ADR-004 — Шесть фиксированных этапов вместо редактора графа узлов

**Решение.** Кейс состоит из **шести фиксированных этапов** (см. DATA_MODEL.md). Редактор
графа узлов **отменён**, зависимость **NodeGraphQt убрана полностью**.

**Обоснование.** Граф узлов — источник переусложнения и scope creep. Фиксированная структура
быстрее в разработке и понятнее преподавателю.

---

## ADR-005 — Ветвление «Вариант B» на этапе 2

**Решение.** На этапе 2 (клинико-эпидемиологический диагноз) есть точка ветвления, но **оба
пути (верный и неверный) проходят одни и те же этапы**. Ошибка **не блокирует** прохождение и
**вскрывается только в финальном отчёте**.

**Обоснование.** Курсант доходит до конца в любом случае (учебная ценность сохраняется), а
разбор ошибок выносится в отчёт.

---

## ADR-006 — Строгий поиск по ключевым словам

**Решение.** Поиск по ключевым словам — **строгое сопоставление** с синонимами, которые задаёт
преподаватель. **Никакого fuzzy-матчинга и устойчивости к опечаткам.**

**Обоснование.** Предсказуемость и контроль преподавателя над тем, что считается верным
запросом; проще в реализации и проверке.

---

## ADR-007 — Механика документов «Вариант B»

**Решение.** Работа с документом состоит из двух частей: (а) курсант **выбирает правильный
документ** из списка, где есть **обманки**; (б) **заполняет поля** документа, которые
**сверяются с ожидаемыми значениями**.

**Обоснование.** Проверяет и узнавание нужного документа, и умение его заполнить.

**Последствия.** Шаблоны документов и поля — см. DATA_MODEL.md и
`.claude/skills/educase-document-templates`.

---

## ADR-008 — Осмотр помещений входит в MVP

**Решение.** Фича «осмотр помещений» (виртуальный обход для санитарно-эпидемиологической
оценки) — **в MVP, не Phase 2**. Двухуровневая навигация: план этажа с кликабельными входами →
интерьерные фото с хотспотами → свободный вывод курсанта, проверяемый по ключевым словам.

**Обоснование.** Анализ показал, что осмотр структурно нужен на этапах 3 и 4 каждого кейса —
значит он часть базовой механики, а не опция.

---

## ADR-009 — Документная модель (без БД)

**Status: Accepted (принято в рабочей сессии, НЕ реконструкция).**

**Решение.** Никакой БД у приложений. Единственный носитель данных — **архив** `.educase`
(кейс) и `.eduresult` (результат). Кейс ведёт себя как документ (открыть → править → сохранить),
библиотека кейсов = папка с файлами в Проводнике (совпадает с ADR-002).

**Обоснование.** Вариант с SQLite добавлял второй слой сериализации (ORM ↔ файл) поверх кодека
архива — лишняя сложность и риск рассинхрона для одного преподавателя/инструмента. Документная
модель убирает целый слой кода (scope-cut в духе принципа №1).

**Последствия.** Из MVP убраны **SQLAlchemy и SQLite**; пакет `infrastructure/db/` удалён.
Сердце инфраструктуры — `infrastructure/archive/`. Развернуть назад — только под библиотеку из
сотен кейсов с поиском по метаданным или общее многопользовательское хранилище (в задаче нет).

## ADR-010 — Без Alembic; версионирование формата в manifest

**Status: Accepted.**

**Решение.** Миграций БД нет (следствие ADR-009). Дисциплина версий — на **формате архива**:
поле `format_version` в `manifest.json` (заложено `FORMAT_VERSION = 1`). При чтении старого
`.educase` кодек распознаёт версию и при необходимости поднимает модель до текущей.

## ADR-011 — Монорепо (общий core + два приложения)

**Status: Accepted.**

**Решение.** Один репозиторий, один `pyproject.toml`, общий `educase_core` +
`educase_constructor` + `educase_player`, два GUI-входа (`gui-scripts`). Каркас собран и проходит
зелёный quality gate.

**Обоснование.** Естественная раскладка для двух приложений с общим доменом и кодеком; без
дублирования.

## ADR-012 — Схема помещений: изображение + хотспоты (НА ПОДТВЕРЖДЕНИИ)

**Status: Proposed — ждёт решения.**

**Предложение.** Преподаватель не рисует план с нуля (нет геометрического редактора). Он
загружает **изображение** схемы/плана объекта и размечает поверх **кликабельные зоны** (входы в
помещения, ячейки личного состава, иконки действий), привязывая к ним фото, контент и ожидаемые
ключевые слова. Осмотр помещений — в MVP (ADR-008), этапы 3–4.

**Обоснование.** Полноценный редактор планов в PySide6 — тяжёлый отдельный инструмент и риск
scope creep (как убранный редактор графа узлов). Модель «картинка + координаты зон» даёт ту же
гибкость дешевле.
