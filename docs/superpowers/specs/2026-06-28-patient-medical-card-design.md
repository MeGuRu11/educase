# N1: медицинские карточки пациентов — дизайн

**Дата:** 2026-06-28
**Статус:** утверждён пользователем
**Область:** Player UI, этап «Пациенты»

## Цель

Привести плитки пациентов и окно первичных данных к виду профессиональной
медицинской карты. На общей странице курсант должен быстро различать пациентов,
а после открытия — удобно читать длинные медицинские записи.

Визуальное направление: вариант A «Клиническая карта».

## Границы

Изменяются:

- `PatientCardWidget`;
- `PatientDetailDialog`;
- стили этих компонентов в `theme.qss`;
- тесты Player UI;
- отметка N1 в `TASKS.md` после завершения реализации.

Не изменяются:

- доменная модель `PatientCard`;
- Constructor;
- формат и кодеки `.epicase`/`.epiresult`;
- содержимое существующих кейсов;
- логика поиска, прохождения и предварительной машинной проверки.

## Плитка пациента

Плитка сохраняет ширину 300 px и участвует в существующей `FlowLayout`.
На лицевой стороне отображаются только:

1. вертикальный teal-маркер с медицинским крестом и подписью «КАРТА»;
2. `PatientCard.title`;
3. подпись «Медицинская карта пациента»;
4. действие «Открыть карту →».

Поля `PatientCard.fields`, ассеты и внутренний `PatientCard.id` на плитке не
показываются.

Вся поверхность плитки активна. Курсор и мягкая teal-подсветка показывают
кликабельность. Плитка получает клавиатурный фокус и открывается по Enter,
Return или Space. При фокусе должна быть видимая акцентная рамка.

Рекомендуемая структура objectName:

- `patientCard` — корневой `QFrame`;
- `patientCardMarker` — цветной маркер;
- `patientCardMarkerSymbol` — медицинский крест;
- `patientCardMarkerText` — подпись «КАРТА»;
- `patientCardTitle` — идентификация пациента;
- `patientCardType` — тип карточки;
- `patientCardAction` — действие открытия.

Все цвета, рамки, размеры шрифтов и состояния задаются в `theme.qss`, без
inline-стилей.

## Окно первичных данных

После активации плитки открывается модальный `PatientDetailDialog`.
Рекомендуемый стартовый размер — около 720 × 560 px, минимальная ширина —
600 px. Содержимое остаётся прокручиваемым.

Структура окна:

1. teal-шапка с подписью «Медицинская карта пациента» и
   `PatientCard.title`;
2. белая область первичных данных;
3. последовательность полей в исходном порядке;
4. при наличии ассетов — секция «Материалы пациента» после полей;
5. нижняя строка с кнопкой «Закрыть».

Каждая пара из `PatientCard.fields` отображается отдельным блоком:

- название поля — малый приглушённый текст;
- значение — основной контрастный текст под названием;
- между блоками — нейтральный разделитель.

Длинные названия и значения переносятся по словам. Значения не обрезаются.
Внутренний UUID пациента не выводится.

Если `fields` пуст, в области данных показывается нейтральное сообщение
«Первичные данные не заполнены». Если `assets` пуст, заголовок секции
материалов не показывается.

Ассеты продолжают отображаться через существующий `AssetImageWidget`.
Отсутствующий или повреждённый ассет использует его текущий безопасный
плейсхолдер; диалог остаётся работоспособным.

Рекомендуемая структура objectName:

- `patientDetailDialog` — диалог;
- `patientDetailHeader` — шапка;
- `patientDetailEyebrow` — тип документа;
- `patientDetailTitle` — идентификация пациента;
- `patientDetailBody` — область данных;
- `patientFieldRow` — блок поля;
- `patientFieldName` — название;
- `patientFieldValue` — значение;
- `patientEmptyState` — пустое состояние;
- `patientMaterialsTitle` — заголовок материалов;
- `patientDetailClose` — кнопка закрытия.

## Данные и зависимости

`PatientsStageView` по-прежнему передаёт в плитку `PatientCard`, а при открытии
диалога — тот же `PatientCard` и загруженный `Mapping[str, bytes]` ассетов.
Новых преобразований, копирования данных или обращений к Infrastructure из UI
не добавляется.

## Тестирование

Тесты на реальных PySide6-виджетах должны проверить:

- плитка показывает заголовок, тип карточки и действие;
- плитка не показывает значения медицинских полей и внутренний id;
- клик испускает `clicked`;
- Enter/Return и Space испускают `clicked`;
- диалог показывает заголовок пациента;
- название и значение каждого первичного поля существуют как отдельные
  элементы;
- длинные значения имеют перенос строк;
- при пустых `fields` отображается нейтральное пустое состояние;
- секция «Материалы пациента» появляется только при наличии ассетов;
- отсутствующий ассет не ломает построение диалога;
- существующая раскладка этапа создаёт нужное число плиток и сохраняет
  ограничение ширины.

После реализации выполняется полный gate:

1. `ruff check src tests`;
2. `mypy src tests`;
3. `pytest -q`;
4. `python -m compileall -q src tests`.

## Живая приёмка

В Player открыть
`C:\Users\user\Desktop\Program\educase_testdata\shigellosis.epicase` и
проверить:

1. восемь плиток выглядят как единый набор медицинских карт;
2. длинные имена не обрезаются и не ломают высоту ряда;
3. медицинские сведения не видны до открытия;
4. карточка открывается мышью и клавиатурой;
5. четыре первичных поля пациента читаются без горизонтальной прокрутки;
6. вертикальная прокрутка работает при большом объёме сведений;
7. закрытие возвращает на тот же этап без потери введённых данных.
