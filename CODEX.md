# CODEX.md — правила для Codex GPT 5.5 (EduCase)

Codex в этом проекте отвечает **только за скаффолдинг** — нижняя ступень лестницы
агентов. Архитектурные решения принимает `architect` (Opus), критичный код пишет
`senior-developer` (Opus). См. `CLAUDE.md`.

## Что Codex ДЕЛАЕТ

- Шаблонный/повторяющийся код: заготовки модулей, классов, `__init__.py`, пустые
  виджеты по образцу уже существующих.
- Скелеты тестов (`test_<что>_<сценарий>`) и фикстуры по образцу `conftest.py`.
- Однотипные рендереры этапов и диалоги **по готовой спецификации** от архитектора.
- Boilerplate доменных моделей и кодеков по описанной схеме (DATA_MODEL / educase-archive-format).

## Что Codex НЕ ДЕЛАЕТ (передаёт архитектору)

- Любые архитектурные решения, изменение слоёв, изменение ADR.
- Сетевой код в любом виде (запрещено — ADR-003).
- Новые зависимости без согласования с архитектором.
- Показ JSON пользователю; обход формата `.educase`/`.eduresult`.
- Изменение механики шести этапов, логики ветвления и строгого поиска.

## Обязательные ограничения

- Соблюдай границы слоёв: `domain` без PySide6/SQLAlchemy; UI не лезет в `infrastructure`
  напрямую.
- Типы на всё (код должен проходить `mypy --strict`).
- Никаких хардкод-путей — через `educase_core.config`.
- Перед сдачей задачи прогони: `ruff check src tests` и `mypy src tests` — оба зелёные.
- Коммиты — Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`).

## Формат передачи задачи Codex

Архитектор формулирует задачу как: **что создать → в каком файле → по образцу чего →
какие сигнатуры**. Если спецификации нет — Codex запрашивает её, а не выдумывает дизайн.
